# 07. 状态机与规则总表

## 7.1 目标

本篇统一整理全系统最重要的业务状态机，作为流程文档和测试设计的共同依据。

## 7.2 ERP 采购明细状态机

### 状态来源

- 主来源：`t_erp_purchase_form_entry.auditstatus`
- 补充来源：`PurchaseFormEntry` 的状态显示逻辑

### 主状态

- 0：草稿或退回
- 1：待审核
- 2：审核通过
- 3：已完成

### 转换规则

- 创建后进入待审核。
- 审核通过后才能进入收货与付款并行链。
- 收货和付款都完成后进入已完成。
- 驳回或撤销收货时会回退。

## 7.3 ERP 调库状态机

### 主状态

- 待审核
- 配货中
- 已发货
- 已完成
- 已驳回或已取消

### 关键规则

- 每个状态推进都会引发库存副作用。
- 取消动作必须按当前所在阶段执行不同的回滚路径。

## 7.4 ERP 组装状态机

### 主状态

- 未提交
- 待组装
- 组装中
- 已完成
- 已终止
- 已作废

### 关键规则

- 组装单先预占子件库存，再允许开始组装。
- 终止和作废都会释放部分资源，但对应触发条件不同。

## 7.5 Amazon 入库计划 V2 状态机

### 主状态

- 1：已创建或已提交
- 2：审核通过
- 3：配货完成
- 4：装箱完成
- 5/6：确认发货阶段
- 7：物流确认
- 8：计划完成
- 11：驳回
- 12：取消

### 关键规则

- 审核通过前不能锁定库存。
- 锁定库存后才允许进入配货。
- 计划状态与货件状态并行存在，不能只看计划主表。
- 异常分支主要是驳回和取消。

## 7.6 Amazon 货件状态机

### 主状态

- 装箱完成
- 物流确认
- 已发货
- 发货执行完成
- 接收中
- 已关闭
- 已取消

### 关键规则

- 部分状态由用户操作推进。
- 部分状态由 Amazon 平台回传推进。

## 7.7 广告报表请求状态机

### 状态维度

- 平台状态 `status`
- 本地处理状态 `treat_status`
- 重试次数 `treat_number`
- 运行锁 `isrun`

### 关键规则

- 报表请求进入轮询队列后，会不断判断平台状态。
- 平台成功并不等于本地处理成功。
- 本地处理失败可进入有限重试。
- 超出时间窗或重试次数后应退出主循环。

## 7.8 广告快照状态机

### 关键规则

- 快照与报表的生命周期相似。
- 快照额外受 `expires` 和较短重试次数约束。

## 7.9 财务凭证状态机

### 主状态

- 草稿或待审核
- 已审核
- 已过账
- 已作废

### 关键规则

- 过账后会回写账簿。
- 期间已关闭时，凭证修改和反过账会受限制。

## 7.10 财务期间状态机

### 主状态

- 未开启
- 已开启
- 已关闭

### 关键规则

- 结账前必须完成前检查。
- 反结账必须保证后续期间满足逆向条件。

## 7.11 报价单状态机

### 主状态

- 1：等待询价
- 2：等待拼团
- 3：等待报价
- 4：已产生报价
- 5：结束

### 关键规则

- 订单状态与订单供应商状态会联动推进。
- 报价确认后，会同步结束订单和供应商关系。

## 7.12 统一建议

- 后续应将 SQL、实体和前端展示中的状态码统一沉淀为集中式状态字典。
- 对于带库存副作用、账簿副作用和外部 API 副作用的状态机，应优先保留“状态 + 动作 + 副作用”三元模型。
